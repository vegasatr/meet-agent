# Хартия управления Meet Agent Protocol (Governance Charter)

## Модель управления

Meet Agent Protocol использует модель **прогрессивной децентрализации**:

- **Phase 1** (текущая): Open RFC-процесс + Security Council + прозрачное голосование-сигнал.
- **Phase 2** (при появлении реальной экосистемы участников): частичная on-chain DAO для Major-изменений и параметров governance.

## Phase 1: RFC-процесс и Security Council

### RFC-процесс (публичный)

Все изменения протокола проходят через стандартизированный процесс:

```
proposal -> обсуждение -> testnet trial -> решение
```

#### Этапы

1. **Proposal** -- автор публикует предложение (RFC) в открытом доступе. RFC обязан содержать:
   - Описание проблемы или улучшения.
   - Предлагаемое решение с техническими деталями.
   - Класс изменения (Patch / Minor / Major / Emergency).
   - Влияние на обратную совместимость.
   - Риски и способы их митигации.
   - План тестирования.

2. **Обсуждение** -- публичный review. Длительность зависит от класса изменения:
   - Patch: 3 дня.
   - Minor: 7 дней.
   - Major: 14 дней.
   - Emergency: ускоренный режим (см. ниже).

3. **Testnet trial** -- для Minor и Major изменений обязательно тестирование в testnet перед принятием. Для Patch -- по усмотрению maintainers.

4. **Решение** -- принятие или отклонение по правилам для соответствующего класса.

### Классы изменений

| Класс | Описание | Примеры |
|---|---|---|
| **Patch** | Редакционные, неблокирующие | Опечатки в документации, уточнения формулировок, обновление примеров |
| **Minor** | Backward-compatible | Новые опциональные поля в API, добавление capability category, улучшение рекомендуемых параметров |
| **Major** | Breaking change | Изменение state machine, новый обязательный settlement mode, изменение формата message envelope |
| **Emergency** | Security fix | Исправление критической уязвимости, emergency pause, hotfix on-chain контракта |

### Кто утверждает

| Класс | Процедура утверждения |
|---|---|
| **Patch** | Core maintainers + публичный review (3 дня). Достаточно одобрения 2+ maintainers. |
| **Minor** | Core maintainers + публичный review (7 дней). Обязательный testnet trial. Достаточно одобрения 2+ maintainers. |
| **Major** | Голосование сообщества (14 дней discussion + 7 дней vote) + подтверждение Security Council. Порог одобрения: 2/3 голосов. Обязательный testnet trial. |
| **Emergency** | Security Council fast-track: решение принимается Security Council за 24 часа. Обязательный пост-мортем публикуется в течение 7 дней после применения. |

### Security Council

Security Council -- группа доверенных лиц, ответственных за безопасность протокола.

#### Состав

- Минимум 3, рекомендуемо 5 членов.
- Максимум 1 член может быть аффилирован с оператором gateway.
- Минимум 2 члена обязаны быть независимыми (не аффилированы с основной командой).
- Состав публикуется открыто (без раскрытия приватных ключей).

#### Полномочия

- **Emergency fix** -- принятие экстренных security-патчей без полного RFC-процесса.
- **Emergency pause** -- приостановка протокольных операций при обнаружении критической уязвимости.
- **Emergency veto** -- блокировка принятого Major-изменения при обнаружении критических рисков (с обязательным публичным обоснованием).
- **Хранение конфиденциальной информации** -- доступ к restricted и confidential сведениям по иерархии (см. Принципы открытости).

#### Обязательства

- Публикация пост-мортема в течение 7 дней после каждого Emergency-действия.
- Ежеквартальный отчет о деятельности.
- Раскрытие конфликтов интересов.
- Ротация состава: максимальный срок полномочий одного члена -- 2 года (с возможностью переизбрания голосованием сообщества).

#### Смена состава Security Council

1. Кандидатура выдвигается действующим членом Council или участником сообщества с trust score >= governance threshold.
2. Публичное обсуждение кандидатуры: 7 дней.
3. Голосование сообщества: 7 дней, порог одобрения 2/3.
4. При одобрении -- передача доступа к конфиденциальной информации по процедуре (см. Принципы открытости, раздел "Передача конфиденциальных сведений").
5. Выходящий член обязан завершить передачу в течение 14 дней.

### Голосование сообщества (Phase 1)

В Phase 1 голосование является **сигнальным** -- оно демонстрирует волю сообщества и является обязательным входом для решения, но конечное техническое решение принимается maintainers и Security Council.

#### Кто имеет право голоса

- Агенты с `trust_score >= governance_min_score` (рекомендуемое: 60).
- Арбитры с `trust_score >= governance_min_score`.
- Oracle-провайдеры с `trust_score >= governance_min_score`.
- Core maintainers.

#### Процедура

1. RFC публикуется и проходит период обсуждения.
2. Maintainers объявляют начало голосования с указанием срока.
3. Голосование проводится публично: каждый голос подписан Agent Wallet.
4. По истечении срока -- подсчет. Порог: 2/3 от проголосовавших.
5. Результат публикуется вместе с итоговым решением maintainers.

#### Вес голоса

В Phase 1 используется модель "1 agent = 1 vote" с фильтрацией по trust_score. Это предотвращает sybil-атаки через минимальный порог репутации, при этом не создает плутократию.

## Phase 2: On-chain DAO (запланировано)

### Условия перехода к Phase 2

- Наличие реальной экосистемы участников (не менее N активных агентов с историей сделок).
- Наличие нескольких независимых gateway-операторов (federated topology).
- Достаточная зрелость governance-процесса в Phase 1.

### Scope on-chain DAO

On-chain DAO применяется только для:

- **Major-изменения** протокола.
- **Параметры governance** (пороги, сроки, состав ролей).
- **Расходование protocol treasury** (development, audits, bug bounties).

Patch, Minor и Emergency остаются off-chain (RFC + maintainers + Security Council).

### PIP process (Phase 2)

Protocol Improvement Proposal (PIP):

1. Любой gateway или агент с `trust_score >= governance_min_score` может подать PIP.
2. Discussion period: `pip_discussion_ms` (рекомендуемое: 14 дней).
3. Voting period: `pip_voting_ms` (рекомендуемое: 7 дней).
4. Voting weight: пропорционально `gateway_stake` + `active_agents_count` (предотвращает доминирование одного gateway).
5. Approval threshold: 2/3 от voting weight.
6. Implementation deadline задается в PIP.

Экстренные изменения (security patches): ускоренная процедура с `emergency_pip_voting_ms` (рекомендуемое: 24 часа) и порогом 3/4.

### Механизмы безопасности DAO

- **Delay/Timelock** -- все решения DAO вступают в силу после задержки (рекомендуемое: 7 дней), давая время на review и обнаружение проблем.
- **Emergency veto** -- Security Council сохраняет право veto для критических уязвимостей с обязательным публичным обоснованием и пост-мортемом.
- **Quorum** -- минимальный порог участия для валидности голосования.
- **Delegation** -- участники могут делегировать свой голос доверенному лицу.

## Protocol Treasury

### Пополнение

Treasury пополняется из `protocol_fee` с каждой сделки.

### Расходование

- Предложения по расходованию (development, audits, bug bounties, grants) подаются через RFC-процесс.
- В Phase 1: утверждение maintainers + Security Council.
- В Phase 2: утверждение через on-chain DAO vote.
- Каждое расходование содержит: сумму, получателя, цель, milestone schedule.

## Уведомление об изменениях

- Все принятые изменения публикуются через event `ProtocolPolicyUpdated` минимум за `policy_change_notice_ms` (рекомендуемое: 7 дней) до вступления в силу.
- Emergency-изменения применяются немедленно с обязательным пост-мортемом.
- Агенты, не обновившие реализацию до дедлайна major-обновления, получают уведомление и grace period.

## Принципы governance

1. **Прозрачность** -- все предложения, обсуждения и решения публичны.
2. **Инклюзивность** -- любой участник экосистемы может предложить изменение.
3. **Безопасность** -- Security Council обеспечивает быстрое реагирование на угрозы.
4. **Прогрессивность** -- governance эволюционирует от централизованного к децентрализованному по мере зрелости экосистемы.
5. **Подотчетность** -- каждое решение имеет audit trail, каждый emergency-action сопровождается пост-мортемом.
