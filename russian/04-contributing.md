# Как вносить изменения в Meet Agent Protocol

## Кто может предлагать изменения

Предложить изменение может любой участник экосистемы:

- Разработчик агента или бота.
- Оператор gateway.
- Арбитр или oracle-провайдер.
- Участник сообщества.

## Процесс внесения изменений

### 1. Определите класс изменения

| Класс | Что включает | Сроки review |
|---|---|---|
| **Patch** | Опечатки, уточнения документации, неблокирующие правки | 3 дня |
| **Minor** | Новые опциональные поля, backward-compatible улучшения | 7 дней + testnet |
| **Major** | Breaking changes, новые обязательные механизмы | 14 дней discussion + 7 дней vote + testnet |
| **Emergency** | Security fix | Fast-track через Security Council (24 часа) |

### 2. Подготовьте RFC (Request for Comments)

RFC -- основной формат предложения. Оно должно содержать:

**Обязательные разделы:**

- **Название** -- краткое описание предложения.
- **Класс изменения** -- Patch / Minor / Major / Emergency.
- **Мотивация** -- какую проблему решает, почему это нужно.
- **Описание** -- техническое описание предложения.
- **Влияние на совместимость** -- ломает ли обратную совместимость, какие агенты затронуты.
- **Риски** -- потенциальные проблемы и способы их митигации.

**Рекомендуемые разделы (для Minor и Major):**

- **Альтернативы** -- какие другие варианты рассматривались и почему выбран этот.
- **План тестирования** -- как предложение будет проверено в testnet.
- **План миграции** -- как существующие агенты перейдут на новую версию.
- **Примеры** -- конкретные примеры использования.

### 3. Опубликуйте RFC

- Разместите RFC в канале сообщества или в соответствующем разделе обсуждений.
- Пометьте класс изменения для правильной маршрутизации.

### 4. Участвуйте в обсуждении

- Отвечайте на вопросы и замечания.
- Вносите правки в RFC на основе обратной связи.
- Будьте готовы к testnet trial для Minor и Major предложений.

### 5. Ожидайте решения

| Класс | Кто принимает решение |
|---|---|
| **Patch** | 2+ core maintainers |
| **Minor** | 2+ core maintainers после testnet trial |
| **Major** | Голосование сообщества (2/3) + подтверждение Security Council |
| **Emergency** | Security Council |

## Специальные случаи

### Emergency: сообщение об уязвимости

Если вы обнаружили уязвимость:

1. **Не публикуйте детали уязвимости в открытом доступе.**
2. Сообщите через защищенный канал Security Council (контактные данные публикуются в канале проекта).
3. Security Council подтвердит получение и определит severity.
4. После закрытия уязвимости -- публикуется пост-мортем.

Мы ценим ответственное раскрытие (responsible disclosure) и планируем bug bounty программу.

### Предложения по governance

Изменения самого governance-процесса (пороги, сроки, состав ролей) проходят как **Major** изменения с голосованием сообщества.

### Предложения по расходованию treasury

Предложения по расходам protocol treasury:

1. Оформляются как RFC с разделами: сумма, получатель, цель, milestone schedule.
2. Проходят стандартный review.
3. В Phase 1: утверждение maintainers + Security Council.
4. В Phase 2: утверждение через on-chain DAO vote.

## Правила поведения

- Конструктивная критика приветствуется.
- Аргументируйте свою позицию техническими фактами.
- Уважайте время reviewers -- подготавливайте полные RFC.
- Не дублируйте существующие предложения, а присоединяйтесь к обсуждению.

## Роли в governance

| Роль | Описание |
|---|---|
| **Contributor** | Любой участник, предлагающий RFC |
| **Core Maintainer** | Ответственные за review и принятие Patch/Minor изменений |
| **Security Council** | Ответственные за Emergency-изменения и безопасность протокола |
| **Voter** | Участник экосистемы с правом голоса (trust_score >= порога) |
