# Принципы открытости Meet Agent Protocol

## Философия

Meet Agent Protocol строится на убеждении, что доверие к протоколу автономных сделок возможно только при его прозрачности. Агенты, управляющие реальными средствами, должны работать по правилам, которые каждый может проверить.

## Что открыто

### Спецификация протокола

Полная спецификация Core MVP публикуется открыто:

- Все state machines (Agent, Intent, Quote, Deal, Escrow, HTLC).
- Правила negotiation, settlement, dispute, finality.
- Canonical terms hash -- формат и порядок полей.
- Message envelope и anti-replay правила.
- Acceptance Rule DSL для machine-readable критериев приемки.
- Settlement Layer: escrow, HTLC swap, milestone, bridge swap.
- API Contract -- полный набор endpoint-ов.

### Governance

- Все изменения протокола проходят через публичный RFC-процесс.
- Голосования по Major-изменениям проводятся открыто.
- Решения Security Council публикуются с обязательным пост-мортемом.
- Protocol Improvement Proposals (PIP) доступны для чтения и обсуждения всем.

### Экономика протокола

- Формула trust score и правила её обновления.
- Policy ценообразования System Agent (Emission Agent).
- Формула эмиссии и правила buyback.
- Fee policy: protocol fee, gateway fee.
- Slashing rules для всех участников.

### Аудит и мониторинг

- Логи сделок System Agent (публичный audit log).
- Matching transparency log.
- Protocol metrics (Prometheus-совместимый формат).
- Ежедневные отчеты по операциям system agents.
- Proof-публикация по каждой сделке.

### Код

- On-chain контракты (Escrow, HTLC, Bundle Coordinator, DelegationAllowance) публикуются open source после аудита.
- Reference implementation SDK публикуется по мере готовности.
- Test vectors для криптографического профиля.

## Что не публикуется

Секретные сведения, ключи и иная информация, способная помочь дестабилизировать протокол или скомпрометировать участников, не публикуются.

Определенная категория информации не публикуется в открытом доступе, так как её раскрытие может быть использовано злоумышленниками:

- Детали реализации конкретных anti-abuse и anti-frontrunning механизмов на уровне gateway runtime (параметры обнаружения, пороги срабатывания, внутренние алгоритмы мониторинга).
- Приватные ключи, seed-фразы, конфиденциальные конфигурации инфраструктуры.
- Конкретные стратегии hot-fix при обнаружении 0-day уязвимостей до момента закрытия уязвимости.
- Внутренние операционные процедуры Security Council до завершения инцидента.

Эта информация классифицируется как **конфиденциальная** и управляется по правилам, описанным ниже.

## Передача конфиденциальных сведений

### Принцип

Конфиденциальная информация хранится ограниченным кругом лиц (Security Council, core maintainers). При необходимости передачи другому лицу действуют строгие правила.

### Правила передачи

1. **Инициация** -- передача конфиденциальных сведений новому лицу возможна только по решению Security Council или по результату голосования сообщества (для случаев смены состава Security Council).

2. **Голосование сообщества** -- в случае, если передача связана с изменением состава ключевых хранителей (например, выход участника Security Council), проводится голосование:
   - Кандидатура нового хранителя выдвигается и обсуждается публично.
   - Голосование проводится среди участников с правом голоса (определяется Governance Charter).
   - Порог одобрения: простое большинство для Patch/Minor-информации, 2/3 для Major-информации.

3. **Процедура передачи**:
   - Передача осуществляется по защищенному каналу (end-to-end encrypted).
   - Факт передачи фиксируется в закрытом audit trail Security Council с указанием: кому, когда, какая категория информации.
   - Передающая сторона подтверждает получение подписью.
   - Предыдущий хранитель обязан удалить свои копии в оговоренный срок (если он покидает роль).

4. **Отзыв доступа** -- при выходе лица из роли хранителя:
   - Все доверенные ключи и credentials ротируются.
   - Факт ротации фиксируется в audit trail.
   - Security Council проводит ревью затронутых систем.

### Иерархия доступа

| Уровень | Кто имеет доступ | Что включает |
|---|---|---|
| Public | Все | Спецификация, governance, экономика, API, открытый код |
| Restricted | Core Maintainers + Security Council | Детали anti-abuse, параметры мониторинга, оперативные процедуры |
| Confidential | Security Council | Приватные ключи инфраструктуры, 0-day hotfix детали, emergency процедуры |

### Аудит доступа

- Журнал доступа к конфиденциальной информации ведется Security Council.
- Ежеквартальный review состава лиц с доступом.
- При обнаружении несанкционированного доступа -- немедленная ротация и пост-мортем.

## Обязательства перед сообществом

1. **Никаких скрытых привилегий** -- System Agent подчиняется тем же протокольным правилам, что и обычные агенты. Его policy публична.
2. **Никакой скрытой эмиссии** -- параметры токена MEET зафиксированы on-chain, ownership revoked.
3. **Публичные proof** -- каждая сделка оставляет верифицируемый proof.
4. **Открытый арбитраж** -- решения арбитров содержат обоснование и ссылки на доказательства.
5. **Прогрессивная децентрализация** -- движение от единого оператора к federated governance и DAO.
